# UI Registry

Living document. Updated after every component is built. Read this before building any new component ‚Äî match existing patterns exactly before inventing new ones.

---

## How to Use

Before building any component:

1. Check if a similar component already exists here
2. If yes ‚Äî match its exact classes
3. If no ‚Äî build it following ui-rules.md and ui-tokens.md, then add it here

After building any component ‚Äî update this file with the component name, file path, and exact classes used.

### ScriptureLinker
`components/layout/ScriptureLinker.tsx` ‚Äî renders nothing (`return null`); mounted once in `app/layout.tsx` alongside `TopNav`. Injects BibleGateway's own `bglinks.js` once, sets `window.BGLinks.version = 'ABTAG2001'`, and calls `linkVerses()` on mount and on every route change (`usePathname()`). The widget auto-detects citation-shaped text anywhere on the page (no wrapper markup needed per-reference) and turns it into a hover link showing BibleGateway's own tooltip ‚Äî used wherever a plain citation is visible, currently `SectionCard`'s Selection item labels. No visual/token footprint of its own since it doesn't render any UI; the tooltip itself is BibleGateway's own styling, not this app's.

### SectionCard
`components/liturgy/SectionCard.tsx` ‚Äî Compile View Section card, matching ui-rules.md's Card spec (`bg-surface border border-border rounded-lg p-6 shadow-[0px_1px_3px_rgba(34,32,28,0.08)]`). Title in `font-serif-display text-[22px] font-semibold leading-[30px]` (same treatment as VerseDisplay's chapter heading). Posture shown as a trailing `*` on standing Sections, nothing for seated. Dynamic-naming Sections (`dynamic_naming: true`) render their leading "Psalm"/"Hymn" word as "Psalm/Hymn" until a real Item disambiguates it ‚Äî e.g. "Psalm/Hymn of Adoration*" (this is a display-only stand-in; a real Psalm/Hymn item-type pass is Feature 21, not yet built). Empty state: `text-sm text-text-muted` "No items yet". `"use client"` as of Feature 08. Takes a full `CompiledSection` plus `liturgyId`/`sectionIndex` (Feature 07), `formulas: Formula[]` (Feature 08), and `prayers: Prayer[]` (Feature 10) ‚Äî header row shows "Add Selection" (`text-sm font-medium text-accent-dark` link to `/reader?liturgyId=...&sectionIndex=...`), "Add Formula" (toggles `AddFormulaPanel`), "Add Verbal Cue" (toggles `VerbalCueForm`), "Add Prayer" (toggles `AddPrayerPanel`), and "Add Sermon" (toggles `SermonForm`, additionally gated on no Sermon item existing yet) **only when each type is present in `section.item_types` (Feature 23, 2026-07-15)** ‚Äî `allowedTypes = section.item_types ?? ALL_ITEM_TYPES`, so a Section that somehow lacks the field (pre-migration data) falls back to showing every button rather than silently losing them. Both templates' `templates.sections` were migrated with a real `item_types` array per Section, derived mechanically from `redesign-plan-v1.1.md` ¬ßY's mapping table ‚Äî substituting `selection` wherever ¬ßY says "Psalm"/"Hymn" (those are display-only stand-ins until Feature 21, not real item kinds yet) and treating "(+ Minister)"/"(+ Guide)"/"(+ Leader/Congregation)" annotations as decorations on the existing kind rather than new kinds (Features 25/27, not yet built). Two Vesper Sections ("The Lord's Table," "Prayer Meeting") get an empty `item_types` array per ¬ßY's "heading only" spec ‚Äî no Add buttons at all. **The whitelist only governs adding new items ‚Äî an item already placed in a Section renders exactly as before even if its type is no longer in that Section's whitelist** (e.g. Morning's real Doxology still shows its pre-existing Verbal Cue, even though Doxology's whitelist is now `["selection"]` only), since Feature 23 is additive-restriction, not a data migration. Verified live against both templates: every Section's visible Add buttons matched the derived mapping exactly, and no pre-existing content disappeared. Both `formulas` and `prayers` arrive as the liturgy's full lists and are filtered client-side to `f.sectionName === section.name` / `p.sectionName === section.name` before being handed to their respective Add panels or picker ‚Äî Section-scoping (2026-07-13 retrofit) is enforced this way rather than a per-Section query, since one fetch already covers the whole Compile View page. Items type-switch on rendering: Selection shows citation (`text-[13px] text-text-secondary`) above text; Formula shows the Formula's name + a shared `LeaderOnlyBadge` (extracted as its own small component in Feature 09, was inline JSX before) when `visibility === 'leader_only'`, above its resolved text (`overrideText` if present, else the *current* `defaultText` looked up live from the passed `formulas` list ‚Äî never frozen, per architecture.md's invariant); Verbal Cue shows a plain "Verbal Cue" label + the same `LeaderOnlyBadge` + an inline "Edit" toggle (`text-[13px] font-medium text-accent-dark`) that swaps the item's display for `VerbalCueForm` pre-filled, calling `updateVerbalCue` on save; Prayer (Feature 10) shows a plain "Prayer" label + the same "Edit" toggle, resolving its text live from the passed `prayers` list by `prayerId` (same live-lookup pattern as Formula's non-override case ‚Äî Prayer has no override concept at all, see `AddPrayerPanel` below) via a small inline `PrayerEditForm` (textarea + "Editing here updates this Prayer in the library for future use." note + Save/Cancel) that calls `updatePrayer` directly on save. All item types share the same liturgical body-text treatment (`font-serif-body text-[17px] leading-[1.75] text-text-primary`), rendered via a small internal `BodyText` component (2026-07-14) that runs the item's resolved text through `lib/text/markdown.ts`'s `parseBoldSegments` and wraps `**bold**` spans in `<strong>` ‚Äî the first place in the app that actually implements the "bold marks a congregational/unison response" convention from founding-days-liturgy-compiler.md ¬ß3, previously documented in ui-rules.md but never implemented. Item label + resolved text now come from the shared `lib/liturgy/resolveItemText.ts` (replacing the inline `resolveFormulaText` helper and duplicated Prayer lookup that lived directly in this file before), and Section titles come from the shared `lib/liturgy/sectionTitle.ts` ‚Äî both extracted specifically so the PDF export (below) can't silently drift from what this component shows on screen.

### LiturgyDocument (PDF export)
`lib/pdf/LiturgyDocument.tsx` ‚Äî Feature 11/12, the shared react-pdf `Document` for both the Leader Guide and Congregation Bulletin, one `audience: 'guide' | 'bulletin'` prop per library-docs.md's pattern, never forked into two templates. Section headings in Old Standard TT (bold, bottom border), body text in Ibarra Real Nova (1.6 line-height) with the same `parseBoldSegments` bold-run handling as `SectionCard`'s `BodyText`. An item is included when `audience === 'guide'` or its resolved `leaderOnly` flag (from `resolveItemText`) is false ‚Äî so a Bulletin generated for a Section whose only item is a leader-only Verbal Cue correctly falls back to "No items yet" rather than an empty gap. Colors come from `lib/pdf/tokens.ts` (literal hex mirroring ui-tokens.md ‚Äî react-pdf has no Tailwind/CSS-variable access, the one necessary exception to "never hardcode hex"). Fonts are registered once via `lib/pdf/fonts.ts`'s `registerPdfFonts()` from real `.ttf` files in `public/fonts/` (Old Standard TT's statics from the `google/fonts` GitHub repo; Ibarra Real Nova only ships as a variable font there, so its regular/bold statics came from Google's font CDN instead, fetched with an old-Android user agent to force a `.ttf` response). Reachable via `GET /api/liturgy/[id]/export?audience=guide|bulletin`, downloaded from two links on the Compile View page ("Download Leader Guide" secondary-button style, "Download Congregation Bulletin" primary-button style).

**Revised for Feature 17 (2026-07-15) ‚Äî same `groupSectionsByPageColumn` fallback as the Compile View page:** when a liturgy's Sections carry `page`/`column` data (Morning), the `Document` renders one A4 `Page` per bulletin page (currently 2) with a `flexDirection: "row"` view of 3 columns each; a `compact` prop on the shared `RenderedSection` sub-component shrinks font sizes (14px‚Üí10.5px heading, 11px‚Üí8.5px body, 9px‚Üí7px item label) since each column is roughly a third of a full page's width. Page 1's first column carries the title/date/Lord's Day text block above its Sections, mirroring the Compile View's placement. When Sections lack `page`/`column` (Vesper ‚Äî no column table exists for it yet), falls back to the original single continuous-column A4 page exactly as before this feature ‚Äî this component and the export route work unmodified for Vesper today even though Vesper has no real PDF use (deferred to v3/v4 per the redesign plan, replaced by `LiturgyWebView` below as of Feature 18; nothing currently prevents hitting the route for a Vesper liturgy id, it just produces a flat-layout PDF no one is meant to link to since the Compile View no longer offers a download link for Vesper). Verified via `pdftotext -layout` against a real Morning liturgy: exactly two "PAGE N" labels, all 18 Sections present in the exact column order `redesign-plan-v1.1.md` ¬ßF specifies, Bulletin audience still correctly excludes the leader-only Pastoral Prayer Verbal Cue ("No items yet" instead).

### LiturgyWebView (public Liturgy Web View, Feature 18)
`components/liturgy/LiturgyWebView.tsx` ‚Äî public, read-only, congregation-facing render of a `CompiledLiturgy`, shareable by URL at `app/liturgy/[id]/view/page.tsx`. Deliberately **template-agnostic** ‚Äî reads the same data shape as `LiturgyDocument`, always renders sections in flat template order (never groups by `page`/`column`, even when present), and filters out leader-only items exactly like the Bulletin PDF audience does (no `LeaderOnlyBadge`, the items are simply omitted rather than shown-and-labeled, since this is meant to be shared publicly). Mobile-first single column: `max-w-[640px] mx-auto px-6 py-10`, left-aligned, no card borders/chrome ‚Äî a plain reading page, not an editing surface (no Add buttons, doesn't import `SectionCard`). Section headings and body text reuse the same `font-serif-display`/`font-serif-body` tokens and `sectionTitle()`/`resolveItemText()`/`parseBoldSegments()` helpers as everywhere else, so it can't drift from how the same content renders in the Compile View or PDF. Verified at 375px mobile width: no horizontal overflow. Verified against a real Morning liturgy (navigated to directly, not yet linked from Morning's Compile View): renders correctly out of the box, including correctly excluding a real leader-only Pastoral Prayer cue ‚Äî confirms the "Morning could gain the same view later" design goal already works today, per `redesign-plan-v1.1.md`'s note, with no component changes needed when that's approved.

On the Compile View (`app/liturgy/[id]/page.tsx`): whenever a liturgy has no `page`/`column` data (Vesper), the "Download Leader Guide"/"Download Congregation Bulletin" buttons are replaced entirely with a single **"View / Share Liturgy"** link (`bg-accent text-accent-foreground`, `target="_blank"`) to this route. **As of 2026-07-15, Morning also gets this same link** ‚Äî shown alongside its two existing PDF buttons rather than replacing them, since Morning's PDF export is real and still the primary output. Extending the link to Morning happened the same day Feature 18 shipped, once the Vesper web view was demonstrated working end-to-end and the template-agnostic design proved out ‚Äî Madrid populated a real Vesper liturgy to check the result, was satisfied, and asked to extend it immediately rather than waiting for a separate feature.

**Real bug fixed in the same pass, unrelated to the column layout:** the PDF title line always showed "Lord's Day {N}" regardless of whether the service date was actually a Sunday ‚Äî the Feature 16 LD#-suppression invariant had only been applied to the Liturgy History list, never to this component or the Compile View page's own header (`app/liturgy/[id]/page.tsx`). Both now compute `isSunday()` and omit the Lord's Day segment entirely for non-Sunday liturgies, matching the invariant everywhere it applies.

### AddPrayerPanel
`components/liturgy/AddPrayerPanel.tsx` ‚Äî inline panel (`bg-surface-secondary border border-border rounded-md p-4`) shown from `SectionCard`'s "Add Prayer" toggle, Feature 10. Receives the Section-filtered `prayers` list plus `sectionName`. ### VerseDisplay
`components/reader/VerseDisplay.tsx` ‚Äî card with wider reading margin (`bg-surface border border-border rounded-lg p-6 md:p-12 shadow-[0px_1px_3px_rgba(34,32,28,0.08)]`). Chapter heading in `font-serif-display text-[22px] font-semibold leading-[30px]`. Verse text in `font-serif-body text-[19.5px] leading-[1.75]` (15% up from ui-tokens.md's 17px base, per Madrid's request for a larger reading size). Highlighted verses get a `bg-{color}-light` span background (all four highlight colors ‚Äî accent, success, info, warning ‚Äî now have a `-light` variant; `warning-light` was added 2026-07-12 to fix the original solid `bg-warning` reading as too dark/brown).

**Two independent click targets, deliberately decoupled (2026-07-12):** clicking verse *text* (the whole span) toggles the personal highlight color via `onVerseClick` ‚Äî completely generic, always available, never touches Section/Selection data. A separate dedicated per-verse marker control (see below) handles Selection-building. Both `verseMarkers`/`onVerseMarkerClick` props are optional and entirely absent in plain Reader browsing (no `liturgyId`/`sectionIndex` in the URL), so the Reader is structurally just a highlighter outside of that context, not merely by convention.

**Verse marker (revised 2026-07-13, three rounds; redesigned again Feature 19, 2026-07-15):** a small element (`w-6 h-6 rounded-sm text-[14px] font-bold` ‚Äî 24px box, 4px radius so it reads as a square with softened corners not a circle) rendered right after the verse number, one per verse, whenever a Section target is active ‚Äî `VerseMarker = { state: "addable" | "pending" | "saved"; label: string }`. **`addable`** (`bg-cta-yellow text-error`, "+" glyph, every verse defaults to this ‚Äî v1.1 yellow-box/red-glyph redesign, was `bg-surface-secondary text-accent`) and **`pending`** (`bg-error-light text-error`, "‚àí" glyph, unchanged by v1.1 per `redesign-plan-v1.1.md` ¬ßH's explicit "not addressed") are still real `<button>`s calling `onVerseMarkerClick`. **`saved`** is no longer a button at all as of Feature 19 ‚Äî `<sup className="text-success text-[13px]">‚óè</sup>`, a passive typographic mark with no `onClick`/`disabled` (was a disabled `bg-success-light text-success-foreground` "‚úì" square button). `VerseDisplay.tsx` branches on `marker.state === "saved"` to choose which markup to render. No visible text on the marker itself ‚Äî the Section name only appears via the native `title` attribute on hover, full (non-abbreviated) text since hover has room. `lib/liturgy/sectionAbbreviation.ts` was deleted since nothing needs the abbreviated form anymore.

**`--color-error-light` added (2026-07-13)** ‚Äî this project had light variants for accent/success/info/warning but not error, needed once the "saved"/"pending" markers moved to the light-tint tier (the `saved` state has since moved off this tier entirely with the Feature 19 sup-mark redesign, but `pending` still uses it). A literal light-tint of `--color-error` lands within a few RGB units of `--color-accent-light` (both are red-family bases), so `error-light` (`#EBD4D2`) is deliberately warmer/more saturated than the other `-light` tokens to stay visually distinct from `accent-light` even at marker size ‚Äî see ui-tokens.md's `@theme` comment for the full reasoning.

**Success confirmation (2026-07-13; repositioned Feature 19, 2026-07-15):** after a Selection saves, a `bg-success-light` bar ("Successfully added to {section}", `text-sm font-medium text-success-foreground`) appears. **As of Feature 19, this now sits below the Citation/Text panel inside the sticky sidebar** (see Reader banner entry below), not above it ‚Äî reversed per `redesign-plan-v1.1.md` ¬ßG. Cleared on chapter change or when a new verse-marker click starts building the next Selection. Caught and fixed a real bug while wiring this up originally: the `useEffect` that resets per-chapter Reader state originally listed `initialHighlights` (an object prop) in its dependency array ‚Äî since `router.refresh()` gives that prop a new reference on every call, saving a Selection immediately re-triggered the effect and wiped the just-set success message before it ever rendered. Fixed by dropping `initialHighlights` from the deps (only `chapter.book`/`chapter.chapter` should trigger the reset) ‚Äî the effect still reads the latest prop value via closure, it just doesn't re-fire on every unrelated refresh.

### Reader "compiling" banner + sticky sidebar (Feature 19, 2026-07-15)
