# UI Registry

Living document. Updated after every component is built. Read this before building any new component — match existing patterns exactly before inventing new ones.

---

## How to Use

Before building any component:

1. Check if a similar component already exists here
2. If yes — match its exact classes
3. If no — build it following ui-rules.md and ui-tokens.md, then add it here

After building any component — update this file with the component name, file path, and exact classes used.

## Components

### ScriptureLinker
`components/layout/ScriptureLinker.tsx` — renders nothing (`return null`); mounted once in `app/layout.tsx` alongside `TopNav`. Injects BibleGateway's own `bglinks.js` once, sets `window.BGLinks.version = 'ABTAG2001'`, and calls `linkVerses()` on mount and on every route change (`usePathname()`). The widget auto-detects citation-shaped text anywhere on the page (no wrapper markup needed per-reference) and turns it into a hover link showing BibleGateway's own tooltip — used wherever a plain citation is visible, currently `SectionCard`'s Selection item labels. No visual/token footprint of its own since it doesn't render any UI; the tooltip itself is BibleGateway's own styling, not this app's.

ALL_ITEM_TYPES`, so a Section that somehow lacks the field (pre-migration data) falls back to showing every button rather than silently losing them. Two Vesper Sections ("The Lord's Table," "Prayer Meeting") get an empty `item_types` array per §Y's "heading only" spec — no Add buttons at all. **The whitelist only governs adding new items — an item already placed in a Section renders exactly as before even if its type is no longer in that Section's whitelist** (e.g. Morning's real Doxology still shows its pre-existing Verbal Cue, even though Doxology's whitelist is now `["selection"]` only), since Feature 23 is additive-restriction, not a data migration. Items type-switch on rendering: Selection shows citation (`text-[13px] text-text-secondary`) above text; Formula shows the Formula's name + a shared `LeaderOnlyBadge` (extracted as its own small component in Feature 09, was inline JSX before) when `visibility === 'leader_only'`, above its resolved text (`overrideText` if present, else the *current* `defaultText` looked up live from the passed `formulas` list — never frozen, per architecture.md's invariant); Verbal Cue shows a plain "Verbal Cue" label + the same `LeaderOnlyBadge` + an inline "Edit" toggle (`text-[13px] font-medium text-accent-dark`) that swaps the item's display for `VerbalCueForm` pre-filled, calling `updateVerbalCue` on save; Prayer (Feature 10) shows a plain "Prayer" label + the same "Edit" toggle, resolving its text live from the passed `prayers` list by `prayerId` (same live-lookup pattern as Formula's non-override case — Prayer has no override concept at all, see `AddPrayerPanel` below) via a small inline `PrayerEditForm` (textarea + "Editing here updates this Prayer in the library for future use." note + Save/Cancel) that calls `updatePrayer` directly on save. All item types share the same liturgical body-text treatment (`font-serif-body text-[17px] leading-[1.75] text-text-primary`), rendered via a small internal `BodyText` component (2026-07-14) that runs the item's resolved text through `lib/text/markdown.ts`'s `parseBoldSegments` and wraps `**bold**` spans in `<strong>` — the first place in the app that actually implements the "bold marks a congregational/unison response" convention from founding-days-liturgy-compiler.md §3, previously documented in ui-rules.md but never implemented. Item label + resolved text now come from the shared `lib/liturgy/resolveItemText.ts` (replacing the inline `resolveFormulaText` helper and duplicated Prayer lookup that lived directly in this file before), and Section titles come from the shared `lib/liturgy/sectionTitle.ts` — both extracted specifically so the PDF export (below) can't silently drift from what this component shows on screen.

### LiturgyDocument (PDF export)
`lib/pdf/LiturgyDocument.tsx` — Feature 11/12, the shared react-pdf `Document` for both the Leader Guide and Congregation Bulletin, one `audience: 'guide' | 'bulletin'` prop per library-docs.md's pattern, never forked into two templates. Section headings in Old Standard TT (bold, bottom border), body text in Ibarra Real Nova (1.6 line-height) with the same `parseBoldSegments` bold-run handling as `SectionCard`'s `BodyText`. An item is included when `audience === 'guide'` or its resolved `leaderOnly` flag (from `resolveItemText`) is false — so a Bulletin generated for a Section whose only item is a leader-only Verbal Cue correctly falls back to "No items yet" rather than an empty gap. Colors come from `lib/pdf/tokens.ts` (literal hex mirroring ui-tokens.md — react-pdf has no Tailwind/CSS-variable access, the one necessary exception to "never hardcode hex"). Fonts are registered once via `lib/pdf/fonts.ts`'s `registerPdfFonts()` from real `.ttf` files in `public/fonts/` (Old Standard TT's statics from the `google/fonts` GitHub repo; Ibarra Real Nova only ships as a variable font there, so its regular/bold statics came from Google's font CDN instead, fetched with an old-Android user agent to force a `.ttf` response). Reachable via `GET /api/liturgy/[id]/export?audience=guide|bulletin`, downloaded from two links on the Compile View page ("Download Leader Guide" secondary-button style, "Download Congregation Bulletin" primary-button style).
